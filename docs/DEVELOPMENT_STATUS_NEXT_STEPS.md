# 开发视角状态清单

## 当前判断

当前项目已经从“存在明显工程硬伤”进入“可以继续做系统性收口”的阶段。

下一步开发不应该继续发散加功能，而应该围绕这三件事推进：

- 稳定性
- 验证闭环
- 主流程可信度

如果这三件事不先做，继续堆功能只会把系统做得更复杂，不会更可靠。

---

## 第一优先级：打穿后端全量测试

### 目标

不是单纯追求测试全绿，而是把剩余失败模块的成因分清楚。

### 为什么优先做

目前只验证了部分后端测试可以通过，例如：

- `settings`
- `rag`

但这不等于后端整体稳定。真正有价值的是把全量后端测试跑完，然后确认：

- 哪些失败是测试基建问题
- 哪些失败是业务代码缺陷
- 哪些失败是外部依赖或环境问题

### 下一步做法

先跑项目自带后端测试入口：

```powershell
python .\AutoTest_Backend\run_backend_tests.py
```

拿到失败清单后，逐个拆成最小化复现，不要在全量测试里盲修。

### 重点怀疑模块

从项目结构和当前修复进度看，下一批更可能出问题的地方是：

- `execution_service`
- `self_heal_flow`
- `database_migrations`
- 部分 repository 层行为

---

## 第二优先级：验证主流程是否真的可运行

### 目标

确认系统不是“局部测试能过，但真实主流程跑不通”。

### 为什么优先做

自动化测试通过不等于系统可演示。

这个项目真正要成立，至少要证明下面这条链路能走通：

- 后端启动
- 前端启动
- 健康检查正常
- 知识库状态可用
- 输入 prompt 后能生成脚本
- 脚本能执行
- 历史记录和统计结果能更新

### 开发重点

这里优先验证的不是 UI，而是系统链路是否真实闭环：

- 前后端是否连通
- Selenium 执行链是否能落地
- 失败后是否真的会进入自愈
- 自愈后的状态与数据是否正确沉淀

如果主流程跑不通，后面再做页面和功能扩展都没有意义。

---

## 第三优先级：收紧执行与自愈模块

### 目标

确保项目核心价值链不是“看起来有逻辑”，而是真实可依赖。

### 为什么优先做

这个项目的真正核心不是前端页面，而是：

```text
生成 -> 执行 -> 失败诊断 -> 自愈 -> 结果沉淀
```

所以开发资源应继续放在后端执行链，而不是优先堆展示层。

### 需要重点检查的点

- 执行状态流转是否一致
- 异常是否完整落库
- 自愈前后策略字段是否准确
- fallback 原因是否可信
- 日志和错误信息是否足够支撑 repair

### 风险

如果这些状态定义本身不干净，后面的历史页、统计页、分析页全部都会建立在脏数据之上。

---

## 第四优先级：把测试基建彻底稳定下来

### 目标

让测试结果具备可信度，而不是靠本机环境“碰巧能跑”。

### 当前已知问题

之前已经确认过，测试运行时目录策略在当前 Windows 环境下会触发权限或卡住问题，这类问题说明测试基建本身曾经不稳定。

### 下一步要做的事

把测试环境假设明确下来：

- 哪些测试依赖文件系统
- 哪些测试依赖 SQLite
- 哪些测试依赖 Chroma
- 哪些测试必须隔离外部服务

### 开发原则

后续不要再接受这两类坏味道：

- 测试依赖本机偶然环境
- 测试通过只是碰巧

---

## 第五优先级：继续清理 RAG 资产质量

### 目标

让 RAG 不只是“能返回结果”，而是“召回内容有语义质量”。

### 当前情况

乱码问题已经修掉一批核心内容，但这只代表文本资产恢复到了“可读”，不代表检索质量已经足够好。

### 下一步要做的事

继续核对这些内容是否一致：

- 知识库文档
- 别名词表
- bilingual prompt 模式
- 搜索/登录/表单等场景文档
- 当前策略逻辑与提示资产之间的对应关系

### 风险

如果知识库内容和生成策略长期漂移，最后表现出来就会像“系统能跑，但生成质量忽高忽低、不可解释”。

---

## 第六优先级：控制前端继续扩张

### 目标

避免在后端闭环没打稳之前，把开发资源消耗在次要层面。

### 当前判断

前端目前已经足够支撑核心流程验证，继续扩页面的收益不高。

### 值得做的前端工作

只建议做两类：

- 支持主流程验证的必要交互
- 修正错误表达，例如状态展示、错误文案、执行详情准确性

### 暂不建议优先投入的方向

- 新增复杂页面
- 重做视觉层
- 做大量非核心交互

前端包体优化可以做，但优先级低于后端稳定性和主流程验证。

---

## 建议执行顺序

1. 跑全量后端测试，列出剩余失败清单。
2. 逐个拆失败原因，先清测试基建，再修真实业务缺陷。
3. 做一次端到端主流程验证，确认系统不是“测试能过但跑不起来”。
4. 再决定是否继续深入修执行链、自愈链或前端展示层。

---

## 一句话结论

下一阶段的开发方向不应该是“继续补功能”，而应该是“证明系统可信”。

如果不先把后端测试和主流程跑通，后续开发只会继续把一个看起来复杂、实际上不稳的系统越做越大。
