# WebUI AutoTest Agent — 项目目标规划、当前状态与下一步计划

> 更新时间：2026-04-23

---

## 一、项目定位与目标

### 1.1 一句话定义

WebUI AutoTest Agent 是一个面向 Web UI 自动化测试场景的智能测试平台，核心能力是打通 **"自然语言需求 → RAG 检索增强 → Selenium 脚本生成 → 本地执行 → 失败诊断 → 自愈重试 → 结果统计"** 的完整闭环。

### 1.2 双重服务目标

| 目标 | 核心关注点 |
| --- | --- |
| **毕业设计答辩** | 技术链路完整、实验可复现、指标可解释、答辩叙事清晰 |
| **求职作品集** | 后端设计能力、前后端联调能力、AI 应用落地能力、工程化实现水平 |

### 1.3 项目亮点

- 自然语言生成 Selenium UI 自动化脚本
- 基于本地知识库的 RAG 增强生成（vector / hybrid / hybrid_rerank 三种检索模式 + BM25 fallback）
- 本地执行测试脚本并记录日志、错误和历史记录
- 首次执行失败后自动触发一轮脚本修复（自愈）
- 展示修复摘要、修复后代码和最终结果
- 统计首次成功率、自愈触发率、自愈成功率和最终成功率

## 二、技术架构

### 2.1 技术栈

| 层级 | 技术 |
| --- | --- |
| 前端 | Vue 3 + Vite + Element Plus + Pinia + Vue Router |
| 后端 | FastAPI + Pydantic Settings + OpenAI Python SDK + ChromaDB + SQLite + Selenium |
| LLM | DeepSeek（OpenAI-compatible SDK） |
| CI | GitHub Actions（Windows runner） |

### 2.2 系统架构

```text
用户
  → 前端工作台（Vue 3 / Element Plus）
  → 后端 API（FastAPI）
  → 生成服务（LLM + Prompt 组装 + RAG 检索）
  → 检索服务（ChromaDB 向量索引 + BM25 fallback + rerank）
  → 执行服务（Selenium 脚本落盘 + Python 子进程执行 + 安全校验）
  → 自愈服务（基于错误日志重新生成修复脚本）
  → SQLite 持久化（测试用例、执行记录、自愈记录、系统设置）
  → 前端展示（历史记录、指标看板、自愈过程）
```

### 2.3 目录结构

```text
webui-autotest-agent/
├── AutoTest_Backend/          # 后端服务
│   ├── app/                   # 核心业务代码

│   │   ├── core/              # 配置、容器、依赖注入

│   │   ├── repositories/      # 数据访问层
│   │   ├── schemas/           # 请求/响应模型

│   ├── docs/knowledge/        # RAG 知识库文档（12 份）

│   │   ├── api/               # API 客户端
```

### 2.4 API 能力清单

| 端点 | 方法 | 用途 |
| --- | --- | --- |
| `/api/v1/health` | GET | 健康检查（后端状态、模型配置、知识库就绪、存储与超时配置） |
| `/api/v1/generate` | POST | 根据用户 prompt 生成 Selenium Python 脚本 |
| `/api/v1/executions` | POST / GET | 创建执行任务 / 查询执行列表（支持分页与状态筛选） |
| `/api/v1/executions/{id}` | GET | 查询单条执行详情 |
| `/api/v1/executions/stats` | GET | 查询统计指标（含趋势桶） |
| `/api/v1/knowledge/rebuild` | POST | 重建知识库索引 |
| `/api/v1/settings` | GET / PUT | 获取/更新运行时配置 |
| `/api/v1/test-cases/{id}` | GET | 查询已生成测试用例详情 |

## 七、风险与限制

| 类别 | 风险 |
| --- | --- |
| **技术风险** | LLM 输出不稳定导致脚本质量波动；页面元素定位易因目标页面变化失效；本地浏览器环境差异影响执行稳定性；自愈成功率依赖错误信息质量 |
| **工程风险** | 本地执行缺少严格安全隔离（非沙箱）；并发执行能力有限；当前更偏单机原型 |
| **研究/展示风险** | 实验样本不足则结论无说服力；只展示成功案例易被追问失败边界；无限制说明易被质疑"泛化能力被夸大" |

---

## 八、核心结论

1. **最缺的不是新功能，而是实验数据和可证明性。** 没有成体系的实验数据，答辩时无法回答生成成功率、自愈成功率等基本问题。
2. **最值得做的不是继续扩技术名词，而是把已有闭环打磨到能经得住追问。** 主流程跑不通，后续所有展示都不可信。
3. **如果目标是毕设和作品集双用途，必须同时补"研究叙事"和"工程证据"。** 只偏向任何一边都不够。

> 下一阶段的开发方向不应该是"继续补功能"，而应该是"证明系统可信"。
