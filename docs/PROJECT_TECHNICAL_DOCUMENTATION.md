# WebUI AutoTest Agent 项目技术文档与目标规划

## 1. 项目概述

### 1.1 项目定位

本项目是一个面向 Web UI 自动化测试场景的智能测试平台，核心目标不是单纯“跑 Selenium”，而是打通以下完整链路：

1. 用户输入自然语言测试需求。
2. 系统基于知识库检索与大模型生成 Selenium Python 脚本。
3. 系统在本地执行脚本并记录日志、错误、运行产物。
4. 当首次执行失败时，系统基于错误信息进行一次自动修复与重试。
5. 前端展示生成结果、执行历史、自愈过程和统计指标。

这类项目如果只做到“生成代码”其实没有太大价值，因为没有闭环。当前仓库真正有意义的部分，是它已经朝“生成 + 执行 + 诊断 + 自愈 + 可视化”这个闭环在走。

### 1.2 当前项目目标

从当前代码和 README 判断，项目主要服务两个现实目标：

1. 毕业设计答辩
2. 求职作品集展示

这两个目标有重合，但重点不同：

- 毕设强调技术链路完整、实验可复现、指标可解释。
- 作品集强调工程实现能力、系统设计能力、AI 应用落地能力。

如果后续规划不区分这两个目标，项目很容易陷入“功能看起来很多，但答辩材料和工程质量都不够硬”的问题。

## 2. 项目现状判断

### 2.1 当前已经具备的能力

结合仓库代码，当前系统已经具备以下核心能力：

- 前后端分离的基础系统架构
- 基于 FastAPI 的后端 API 服务
- 基于 Vue 3 + Element Plus 的前端工作台
- 基于本地知识库的 RAG 检索增强
- 基于 OpenAI-compatible SDK 的脚本生成
- Selenium Python 脚本本地执行
- 执行日志、错误信息、脚本路径、运行目录持久化
- 一轮自愈修复与重试
- 执行历史、运行状态、自愈记录、统计指标展示
- 运行时配置管理，如超时、自愈次数、并发数

### 2.2 当前项目的真实阶段

这个项目不是“从 0 到 1 的概念验证”，而是“已经完成可演示 MVP，但还没达到高质量毕业设计/可投递作品集的完成态”。

更直接一点：

- 它已经能演示。
- 它还不够强，尤其在实验严谨性、鲁棒性、前端测试、可扩展性、真实案例覆盖上仍然偏弱。

如果你把现在这份项目直接当成“完整毕业设计终稿”，标准是不够的；如果把它当成“结构已经成型、接下来重点补硬指标和工程证据”的版本，这个判断才成立。

## 3. 技术架构

### 3.1 总体架构

```text
用户
  -> 前端工作台（Vue 3）
  -> 后端 API（FastAPI）
  -> 生成服务（LLM + Prompt 组装）
  -> 检索服务（本地知识库 + ChromaDB / BM25 fallback）
  -> 执行服务（Selenium 脚本落盘、本地 Python 子进程执行）
  -> 自愈服务（基于错误日志重新生成修复脚本）
  -> SQLite 持久化（测试用例、执行记录、系统设置）
  -> 前端展示历史、指标、自愈过程
```

## 4. 后端设计

### 4.1 后端职责

后端承担的不是单一“接口层”，而是整个业务闭环的调度中心，主要职责如下：

- 管理系统配置
- 维护知识库索引
- 生成测试脚本
- 创建执行任务
- 执行脚本并采集日志
- 在失败后发起自愈修复
- 统计执行结果与趋势

#### 生成服务

文件：

- [AutoTest_Backend/app/services/generation_service.py](D:/GitHub_repo/00_active/webui-autotest-agent/AutoTest_Backend/app/services/generation_service.py)

职责：

- 接收自然语言提示词

- 调用 RAG 服务检索知识
- 组装增强 Prompt
- 调用 LLM 生成 Selenium Python 脚本
- 清洗代码并持久化测试用例

它做得比较对的一点是：不是直接把用户 prompt 丢给模型，而是加了知识上下文、任务意图提示和输出约束。这至少说明项目在往“可控生成”方向走，不是纯碰运气。

#### 检索服务

文件：

- [AutoTest_Backend/app/services/rag_service.py](D:/GitHub_repo/00_active/webui-autotest-agent/AutoTest_Backend/app/services/rag_service.py)

职责：

- 从本地知识库文档构建索引
- 支持 `vector`、`hybrid`、`hybrid_rerank` 三种检索模式
- ChromaDB 不可用时退化为 BM25/fallback 方案
- 输出上下文片段、来源文档和命中数量

这里的价值不在“用了 RAG”这个词，而在于：

- 有本地知识库
- 有不同检索模式
- 有 rerank 逻辑
- 有 fallback 机制

这比“简单把知识文档拼到 prompt 后面”要更像一个真实系统。

#### 执行服务

文件：

- [AutoTest_Backend/app/services/execution_service.py](D:/GitHub_repo/00_active/webui-autotest-agent/AutoTest_Backend/app/services/execution_service.py)

职责：

- 校验生成代码安全性
- 创建执行记录
- 启动后台线程异步执行任务
- 将脚本写入运行目录
- 通过 Python 子进程执行 Selenium 脚本
- 记录 stdout/stderr、错误、状态
- 在失败后触发自愈修复

它包含几个关键设计点：

1. 有代码安全校验
2. 有执行队列并发限制
3. 有运行目录隔离
4. 有超时控制
5. 有失败后的单轮自愈机制

这几项是系统工程化的关键，不是“能运行”就自然会有的。

### 4.3 代码安全机制

执行前会通过 AST 做静态校验，限制：

- 非白名单 import
- `eval`、`exec`、`open`、`input` 等危险调用
- `os.`、`subprocess.`、`socket.`、`pathlib.` 等危险前缀调用

这说明项目已经意识到“LLM 生成代码不能直接裸跑”的风险。

但要明确一点：这只是基础防护，不等于真正安全沙箱。当前系统仍然是在本机 Python 环境执行 Selenium 脚本，风险被降低了，但没有被消除。

### 4.4 数据持久化

从模型定义看，后端至少维护以下核心实体：

- `TestCaseRecord`
- `ExecutionRecord`
- `SelfHealAttemptRecord`
- `SettingRecord`

文件：

- [AutoTest_Backend/app/models/entities.py](D:/GitHub_repo/00_active/webui-autotest-agent/AutoTest_Backend/app/models/entities.py)

其中关键数据包括：

- 测试用例标题、原始 prompt、生成代码、RAG 上下文
- 执行状态、运行目录、脚本路径、日志、错误、校验错误
- 自愈次数、是否触发自愈、修复前后策略、自愈代码
- 系统运行时设置

这类数据结构已经足够支撑演示、复盘、统计和实验对比。

## 6. 核心业务流程

### 6.1 脚本生成流程

```text
用户输入测试需求
  -> 后端校验 prompt
  -> 策略分析
  -> RAG 检索知识片段
  -> 拼接增强 Prompt
  -> 调用 LLM 生成 Selenium 代码
  -> 清洗代码
  -> 保存测试用例
  -> 返回前端展示
```

### 6.2 脚本执行流程

```text
前端提交执行请求
  -> 后端创建执行记录
  -> 安全校验生成代码
  -> 后台线程启动执行
  -> 落盘生成脚本
  -> Python 子进程执行
  -> 记录 stdout/stderr 与状态
  -> 成功则结束
  -> 失败则进入自愈流程
```

### 6.3 自愈流程

```text
首次执行失败
  -> 读取错误信息与日志
  -> 分析修复策略
  -> 调用 LLM 修复脚本
  -> 再次执行修复后脚本
  -> 记录修复摘要、修复代码、修复状态
  -> 输出最终状态
```

### 6.4 知识库流程

```text
知识文档目录
  -> 文档切分
  -> 向量索引 / BM25 数据准备
  -> 用户查询时多路检索
  -> 结果融合 / rerank
  -> 返回上下文片段与来源
```

## 8. 项目优势与不足

### 8.1 当前优势

- 业务闭环完整，不是单点 Demo
- 有 RAG，不是裸 Prompt 生成
- 有执行与自愈，不是只停留在代码生成
- 有前端可视化，便于演示
- 有测试与运行脚本，具备基础工程化能力
- 有运行时设置，具备一定可控性

### 8.2 当前主要不足

#### 不足 1：真实实验样本不足

现在最容易出问题的不是代码跑不起来，而是缺少足够多、足够真实的实验场景来证明系统有效。

没有成体系的实验数据，答辩时你很容易被追问：

- 生成成功率是多少？
- 自愈成功率是多少？
- 对哪些页面类型有效？
- 对哪些页面类型明显无效？

这些问题不能靠口头解释，需要数据。

#### 不足 2：自愈机制还比较浅

当前自愈主要是“一轮错误驱动修复”。这能演示，但还算不上强鲁棒系统。

限制包括：

- 自愈轮次少
- 修复策略还不够结构化
- 缺少错误分类统计与针对性策略库

#### 不足 3：执行隔离性有限

当前脚本是在本地 Python 子进程中执行，虽然有 AST 校验，但本质上不是严格沙箱。

这意味着：

- 更适合本地研究和演示
- 不适合直接作为多人共享平台上线

#### 不足 4：前端自动化验证偏弱

仓库里已有部分前端 store/view-model 测试，但还缺浏览器级集成验证。

这会导致一个问题：

- 业务逻辑可能是对的
- 但实际页面交互层未必稳定

#### 不足 5：缺少更清晰的“成果证明材料”

对于毕设/求职来说，仅有代码仓库不够，还需要：

- 实验设计
- 指标统计
- 案例对比
- 失败案例分析
- 架构图与流程图

这些材料现在还没有完全沉淀成体系。

- 知识库可重建
- 可以完成生成、执行、自愈、历史、指标展示

验收标准：

- 至少 3 个场景可完整演示
- 无明显阻断级错误

### 里程碑 2：可答辩版本

目标：

- 补齐实验数据
- 补齐技术文档
- 补齐答辩材料

验收标准：

- 有结构化实验表格
- 有指标图
- 有成功/失败案例对比
- 有系统限制说明

### 里程碑 3：可作为作品集版本

目标：

- 增强前端观感
- 强化工程质量
- 补充测试和案例

验收标准：

- 仓库结构清晰
- README 和 docs 完整
- 演示链路稳定
- 能体现系统设计能力，不只是 API 拼接

## 11. 建议的优先级排序

建议按下面顺序推进，而不是继续随机加功能：

1. 稳定已有主链路
2. 补实验数据和指标
3. 补技术文档和答辩材料
4. 扩展高价值场景
5. 优化前端展示
6. 最后再考虑高级增强项

原因很简单：

- 没有实验数据，项目说服力不足。
- 没有稳定链路，所有展示都不可信。
- 没有文档和叙事，老师和面试官都只能靠猜。
- 过早做高级功能，大概率是自我感动，不是有效交付。

## 12. 风险与限制

### 12.1 技术风险

- LLM 输出不稳定，脚本质量存在波动
- 页面元素定位容易因目标页面变化而失效
- 本地浏览器环境差异会影响执行稳定性
- 自愈成功率依赖错误信息质量

### 12.2 工程风险

- 本地执行缺少严格安全隔离
- 并发执行能力有限
- 当前更偏单机原型，尚未达到生产级平台要求

### 12.3 研究与展示风险

- 若实验样本不足，结论不具备说服力
- 若只展示成功案例，老师很可能追问失败边界
- 若没有限制说明，项目容易被质疑“泛化能力被夸大”

## 13. 结论

这个项目当前最合理的定义不是“已经做完”，而是“核心闭环已成型，下一步应聚焦证据、稳定性和材料沉淀”。

最关键的判断有三个：

1. 现在最缺的不是新功能，而是实验数据和可证明性。
2. 现在最值得做的不是继续扩技术名词，而是把已有闭环打磨到能经得住追问。
3. 如果目标是毕设和作品集双用途，必须同时补“研究叙事”和“工程证据”，只偏向任何一边都不够。

## 14. 下一步执行清单

建议你接下来直接按这个清单推进：

1. 固化 10 到 20 个标准实验场景。
2. 跑出每个场景的首次执行结果和自愈结果。
3. 生成一份实验结果总表和统计图。
4. 补充失败案例分析文档。
5. 清理主链路中的阻断问题。
6. 完善 README 与 `docs/` 文档集合。
7. 准备答辩 PPT 所需架构图、流程图、指标图。

如果你不按这个顺序，而是继续堆功能，结果大概率是代码更多，但论文、答辩和展示反而更弱。
