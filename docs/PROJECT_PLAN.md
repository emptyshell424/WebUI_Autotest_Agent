## 阶段一：工程基础设施修复（预计 2-3 天）

目标：让项目达到"可构建、可验证、可复现"的最低工程标准。

### 任务 1.1 修复前端生产构建

- **问题**：`npm run build` 在 Windows 下因 `spawn EPERM` 失败
- **根因**：`run-build.mjs` 使用 `node scripts/run-build.mjs` 方式调用 Vite，在 Windows 下触发子进程权限问题
- **方案**：
  - 方案 A：将 `package.json` 的 `build` 脚本改为 `vite build`，直接使用标准 Vite CLI（`vite.config.js` 已有完整配置，`run-build.mjs` 的 manualChunks 配置与 `vite.config.js` 重复）
  - 方案 B：若需保留 `run-build.mjs` 的特殊逻辑，改用 `cross-env` 或调整 Node 子进程启动方式
- **验收**：`npm run build` 在 Windows 下成功产出 `dist/` 目录，无报错
- **优先级**：P0

### 任务 1.2 端到端主流程验证

- **目标**：确认系统不是"测试能过但跑不起来"
- **验证链路**：
  1. 后端启动 → `http://127.0.0.1:8000`
  2. 前端启动 → `http://127.0.0.1:5173`
  3. Settings 页健康检查正常
  4. 知识库重建成功
  5. Workbench 输入 prompt → 生成脚本
  6. 执行脚本 → 记录结果
  7. History 页确认记录更新
  8. Metrics 页确认统计更新
- **验收**：上述 8 步全部走通，无阻断错误
- **优先级**：P0

### 任务 1.3 稳定后端测试基线

- **目标**：确认 30/30 测试在干净环境下可复现
- **做法**：
  - 在新终端执行 `python AutoTest_Backend/run_backend_tests.py`
  - 如有失败，按模块拆分原因（测试基建 vs 业务缺陷 vs 外部依赖）
  - 补充 `execution_service` 和 `self_heal_flow` 的边界测试
- **验收**：全量测试通过且可重复
- **优先级**：P0

## 阶段二：实验数据与证据沉淀（预计 3-5 天）

目标：形成答辩可用的实验体系，让项目从"能演示"升级为"能证明"。

### 任务 2.1 固化标准实验场景

- **当前**：`EXPERIMENT_SCENARIOS.md` 有 15 个场景建议，但均为 Pending
- **做法**：
  - 从 15 个场景中选取 10-12 个作为正式实验集
  - 每个场景明确：输入 prompt、目标页面、预期行为、判定标准
  - 分三组：首次生成场景（F-01~F-05）、复杂交互场景（C-01~C-03）、自愈场景（H-01~H-03）
- **交付物**：更新 `EXPERIMENT_SCENARIOS.md`，每个场景含完整定义
- **优先级**：P0

### 任务 2.2 执行首批手工实验批次

- **前提**：阶段一的主流程验证通过
- **做法**：
  - 选择稳定靶场（`vue-admin-template` 本地启动 或 公开测试站点）
  - 逐个执行 F-01 ~ F-05，记录：
    - 输入 prompt 原文
    - 生成脚本摘要
    - 首次执行结果（成功/失败+错误信息）
    - 是否触发自愈
    - 自愈后结果
    - 最终状态
  - 每个场景截图：输入界面、执行结果、历史记录、指标页
- **交付物**：更新 `EXPERIMENT_RESULTS.md`，F-01~F-05 状态从 Pending → Verified
- **优先级**：P0

### 任务 2.3 执行自愈实验批次

- **做法**：
  - 对 H-01 ~ H-03 场景，构造可控的首次失败（如故意使用错误选择器、过短等待时间）
  - 记录修复前代码、错误信息、修复后代码、最终结果
  - 截图对比：首次失败界面 vs 自愈成功界面
- **交付物**：更新 `EXPERIMENT_RESULTS.md`，H-01~H-03 状态从 Pending → Verified
- **优先级**：P1

### 任务 2.4 汇总实验指标

- **做法**：
  - 汇总所有已完成场景的统计指标：
    - 首次成功率
    - 自愈触发率
    - 自愈成功率
    - 最终成功率
  - 按场景类型分组统计
  - 生成指标图表（可用 Markdown 表格 + 简单 ASCII 图）
- **交付物**：实验结果总表 + 指标汇总
- **优先级**：P1

### 任务 2.5 失败案例归因分析

- **做法**：
  - 对每个失败场景（首次失败且自愈也失败），分析根因：
    - 选择器失效 → 页面结构变化 / 动态 ID
    - 等待不足 → 异步加载时序
    - 逻辑错误 → 生成代码断言条件错误
    - 其他
  - 形成失败模式分类表
- **交付物**：失败案例分析文档
- **优先级**：P1

- **验收**：核心场景的 RAG 召回片段与查询语义相关，无高噪声 chunk 被选中

## 阶段四：执行编排与自愈增强（预计 3-4 天）

目标：让核心价值链从"看起来有逻辑"升级为"真实可依赖"。

### 任务 4.1 执行任务取消能力

- **当前**：执行使用 `threading.Thread(daemon=True)`，无法从外部取消
- **方案**：
  - 引入 `threading.Event` 作为取消信号
  - 子进程执行时通过 `process.terminate()` 停止
  - 新增 `DELETE /api/v1/executions/{id}` 端点
  - 前端 History 页增加"取消"按钮（仅对 running 状态生效）
- **验收**：可取消正在执行的脚本，状态正确流转为 `cancelled`
- **优先级**：P1

### 任务 4.2 执行状态流转一致性检查

- **做法**：
  - 梳理所有合法状态转换：`queued → running → success / failed / cancelled`
  - 自愈流程中的状态：`failed → healing → healed_success / heal_failed`
  - 检查 repository 层是否正确约束状态转换
  - 补充状态转换的单元测试
- **验收**：无非法状态转换，所有转换路径有测试覆盖
- **优先级**：P1

### 任务 4.3 自愈策略分类增强

- **当前**：自愈是"一轮错误驱动修复"，策略较浅
- **做法**：
  - 对常见失败类型分类：选择器失效、等待超时、iframe 切换、弹窗处理、断言错误
  - 在 `strategy_service` 中为每类失败生成针对性修复提示
  - 记录修复前后策略字段（`repair_strategy_before` / `repair_strategy_after`）
- **验收**：自愈日志中可见分类策略，不同错误类型触发不同修复提示
- **优先级**：P2

### 任务 4.4 异常落库完整性

- **做法**：
  - 检查所有执行异常是否完整写入 `ExecutionRecord`（error 字段非空、日志非截断）
  - 检查自愈记录的 fallback 原因是否可信
  - 补充缺失的异常捕获点
- **验收**：执行失败时，历史页可看到完整错误信息和日志
- **优先级**：P1

## 风险与依赖

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| DeepSeek API 不稳定或额度不足 | 实验批次无法按时完成 | 提前准备备用 API Key，或切换到其他 OpenAI-compatible 模型 |
| Windows 环境下 `spawn EPERM` 无法根治 | 前端构建持续受阻 | 考虑在 WSL 或 CI 环境中构建，本地仅做 dev 验证 |
| 靶场页面不稳定（动态 ID、频繁改版） | 实验结果不可复现 | 优先使用本地靶场 `vue-admin-template`，避免依赖外部站点 |
| LLM 生成质量波动 | 实验指标不稳定 | 每个场景至少跑 3 次，取众数结果 |
| 时间不足，无法完成全部阶段 | 答辩材料不完整 | 严格按优先级执行，P0 必须完成，P2 可裁剪 |
